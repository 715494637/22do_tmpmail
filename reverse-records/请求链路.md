# 请求链路

- 目标请求：`/action/mailbox/login`、`/zh/inbox/#/{email}`、`/zh/content/{messageId}`、`/action/mailbox/download`
- 目标对象：临时邮箱生成、收件列表、单封邮件内容
- 当前样本状态：已确认，正常游客态
- 关键未闭环：`/action/mailbox/applyToken` 的 `cfToken` 仍依赖浏览器 Turnstile；但当前任务的最小可用链路不需要走这条 API
- 样本编号：`S-001`
- 证据编号：`E-001`

| 项目 | 内容 |
|---|---|

## 辅助链路F-轮询 API

| 项目 | 内容 |
|---|---|
| 接口 | `/action/mailbox/applyToken` -> `/action/mailbox/message` |
| 触发方式 | 收件箱页 `DOMContentLoaded` 后自动执行 |
| 上游请求 | FingerprintJS `uuid` + Turnstile `cfToken` |
| 响应结果 | `JWT` -> 增量轮询新邮件 |

说明：

- 这条链路是浏览器增量轮询链，不是当前任务的最小可用链。
- 真实浏览器已取证到 `uuid`、`cfToken`、`JWT`、`lastime`，但 Python 直连复放会被 Cloudflare 拦截。
- 采用 `curl_cffi` 浏览器 TLS 指纹后，页面链 `请求A -> 请求B -> 请求C -> 请求D -> 请求E` 已可纯请求稳定跑通，无需复用浏览器 cookie。

## Runtime Update 2026-03-21

- 已证实一阶分歧：
  从真实页面 `getToken()` 取到的一枚 fresh `cfToken`，在同一浏览器页内调用 `/action/mailbox/applyToken` 会成功；
  把同一枚 fresh `cfToken` 交给 `curl_cffi` 首次复放，请求返回 `{"status":false,"msg":"Invalid cfToken!"}`。
- 结论：
  `cfToken` 不是可脱离环境搬运的纯字符串参数，而是绑定在 Turnstile / Cloudflare 的真实浏览器运行态上。
- 最小依赖集：
  纯请求最小业务链仍然是 `gmail -> login -> inbox -> content -> download`。
  如果必须调用 `/action/mailbox/message`，就需要先在真实 inbox 页里执行 `initToken()` 取 JWT，再由 Python 复用该 JWT 发请求。
